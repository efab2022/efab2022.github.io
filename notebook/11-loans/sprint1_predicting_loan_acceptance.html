Machine Learning - Making predictions¶

Predicting Loan Eligibility¶

The first iteration requires us to fit a model that marks which loans should be accepted and which ones should be rejected. This target column does not exist in our dataset, so we will have to create it. This requires us to select/define a "success" criteria for our loans.

This newly created column will contain a result (0 or 1) that indicates whether we should have accepted that given loan or not. This will be derived from other data in our dataset. The data used to derive this new target column will be dropped so the model cannot easily find the pattern that we used to create it. We want our model to find the patterns in "the rest of the remaining data" (the data that we would get at the beginning of the loan application, not at the end).

Possible outcomes:

  • In the best case scenario (eg. if I was able to identify a pattern and to created a target column properly), our model should be able to fit our data relatively easily.
  • In the worst case scenario (eg. the criteria I used was terrible or just random), expecting our model to fit it properly, would result in very strong overfitting.
  • Let's hope we are closer to the first case. The tests below will help us determine how good was I at creating this target column.

Implementation details: The creation of this column was already done as one of the first cleaning steps. Check the loans_clean.FillTarget() Transformer. Unfortunately for us, I am not a domain expert in loans, and our resident financial analyst is out on holidays... so the criteria I decided to use to define this target column is flaky (at best) or completely nonsensical and random (at worst). I do not attach my self esteem to my ability to perform advanced financial analysis :)

Best of luck! Let's do it!

Checking XGBClassifier with Optuna Hyperparameter Tuning¶

In this next section, we will pick the best performing model from the previous iteration and we will fine tune it using Optuna instead of BayesianCV

Some of the challenges I wanted to explore in this section were:

  • How to use Optuna for hyperparameter tuning
  • How to use Optuna's in a simulated "distributed" computing environment
    • to learn the tool
    • to reduce the training time
    • to better utilize my laptop's CPU
    • to practice with multi-thread/multi-process programming

The lessons learned:

  • Using Optuna's "easy parallelization" is pretty easy, and actually does what it says... you can easily spin up multiple processes and the CPU utilization will increase almost linearly (until you get more processes than CPU cores).
  • It was relatively easy to write a wrapper (check optuna_utils.py) that handles all of the parallel optuna work, and to leave the problem-specific objective function in the notebook.
  • Everything scales well until you try to use too much data for training. And not for the reason you'd imagine!
    • I will leave the tech lessons learned for the summary, but suffice to say, there is a very good reason why the code below can only use 14k rows.
    • PS: future Edu following up: Turns out, the consistent crashes got fixed the following day. It's unclear what caused the issue, or what fixed it, but the crashes were so consistent that I was able to develop and document multiple rounds of experiments (see technical notes at the end).

from sklearn.model_selection import KFold
from sklearn.metrics import f1_score
import optuna


# This is a quick and dirty version of the function. A more long-term
# version will be created in `optuna_utils.optuna_optimize_model()` with long-term improvements.
# That one is the function that I will use for the rest of this project, and for
# future projects (if you see bugs/improvements for this one),
# check if the one in optuna_utils fixes it :)
def optuna_optimize_model(trial):
    """
    generates a new candidate from the provided trial config and
    evaluates their performane using 5 fold CrossVal.
    Default to using 5 fold, and F1 Score to measure performance.
    """
    model = optuna_utils.XgbClassifierCandidateProvider().provide(trial)

    kf = KFold(n_splits=5)
    scores = []

    X = X_df.to_numpy()
    y = y_df.to_numpy()

    for train_index, val_index in kf.split(X):
        X_train_kf, X_val_kf = (
            pd.DataFrame(np.array(X)[train_index], columns=X_df.columns),
            pd.DataFrame(np.array(X)[val_index], columns=X_df.columns),
        )
        y_train_kf, y_val_kf = (
            pd.Series(np.array(y)[train_index]),
            pd.Series(np.array(y)[val_index]),
        )
        model.fit(X_train_kf, y_train_kf)
        y_pred = model.predict(X_val_kf)
        score = f1_score(y_val_kf, y_pred)
        scores.append(score)

        # let's also try some Pruning action, to see if it speeds us up :)
        intermediate_result = -(sum(scores) / len(scores))
        trial.report(intermediate_result, step=len(scores))
        if trial.should_prune():
            raise optuna.TrialPruned()

    avg_score = -(sum(scores) / len(scores))
    return avg_score

A couple of good things about this method:

  • Saves time on future runs:
    • Even if the model was not persisted to disk using pickle (which it is), we'd still be able to access the parameters that our study encountered, even from day to the next; saving us hours of computation time.
  • Saves time during training:
    • We can hyperparameter tune over different days, because we persist our trials.
    • We prune trials that do not show promising result, early! Saving valuable CPU and Dev time.
  • Resumeable training means we don't have to do everything in one sitting:
    • We can tune a bit, kill it early and see if our model works well enough. If it doesn't, we can resume trials where we left them without having to restart.
  • We use our fast_xgb() shortcut to make sure that our model (after tuning) uses all CPUs (tuning is done with 1 core per trial, but that's suboptimal for training/predicting)
