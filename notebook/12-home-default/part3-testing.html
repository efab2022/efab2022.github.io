Model deployment¶

This model requires data from a variety of places:

  • current application data
  • historical data (credit/balance/point of sale/etc...)

Some of these pieces of data are only available "right now" when the customer requests the information for a credit/load, but the historical ones are available well before that.

Options Available¶

Option 1 - Create a Stateless service¶

Option One would be to create a stateless service, and then we require the requestor/HTTP client to provides us all of the data that we will need to make predictions.

Not great:

  • cumbersome to implement
    • high dependency on pipelines and transformations. Any time changes in terms of preprocessing, the server must be changed or trigger a reload of cleaning pipelines.
  • Heavy payloads
    • client must provide all of the data (some of the data might already be available to us from more trustworthy sources)
  • Brittle contracts/fields
    • changes to fields/fieldnames/types/etc...
  • Potentially less secure
    • potentially vulnerable to data poisoning/injection from malicious actors.

Option 2 - Query a Feature Store¶

Option Two is to use a different paradigm: a Feature Store to store customer-historical data and then the requestor only needs to provide data that we don't have yet (the last bit of data that they submit when they submit an application).

Benefits:

  • HTTP Client/Requestor only needs to provide the last bits of data we do not yet have (for example):
    • amt_income_total
    • amt_credit
    • amt_annuity
    • amt_goods_price
    • days_birth
    • days_employed

(we're imagining that the historical data is collected weekly/daily but that we don't have some details related to the application)

Creating our Feature Store¶

For our feature store, we could have a precomputed set of features for our clients and we retrieve the relevant info for our client when a new request comes in.

Data that we expect in client requests:

  • amt_income_total
  • amt_credit
  • amt_annuity
  • amt_goods_price
  • days_birth
  • days_employed
  • credit_annuity_ratio (not required to be provided, we will calculate it ourselves)

Data we will retrieve from the Feature Store:

  • all the other data (things that do not change frequently and that we can precompute before the client even comes to us)

This list could be debated and adjusted as needed, this is just a simplified example.

We have carefully selected these feature so that the process is a bit more complex than the most trivial code example, but without making it unnecessarily convoluted.
